Product Perspective

What Changed for Users

How fine-tuning improved day-to-day Robot Framework automation assistance — from code quality to safe boundary handling.

+70%
The model now knows what it doesn't know.
When a user asks about something outside Robot Framework scope, the fine-tuned model adds a disclaimer or appropriately declines — 80% of the time, up from 10%. Users get honest, scoped answers instead of confidently wrong ones.

Before vs. After Fine-Tuning

Base Model (Before)
  • Generates Robot Framework syntax incorrectly ~59% of the time
  • Answers out-of-scope questions confidently with no disclaimer (90% of cases)
  • Follows task instructions only 22% of the time
  • No domain specialization — generic LLM responses
Fine-tuned Model (After)
  • Generates correct Robot Framework syntax 48% of the time (+7.4pp)
  • Appropriately declines or disclaims out-of-scope questions 80% of the time
  • Follows task instructions 28% of the time (+6pp)
  • Specialized for RF automation: keywords, test cases, libraries

Key Use Cases Improved

Writing Test Cases
The model now reliably produces *** Test Cases *** table structure, keywords, and correct spacing — the most common RF authoring task.
Keyword Libraries
Custom keyword definitions with *** Keywords *** blocks are generated with proper indentation and argument handling.
Scope Boundaries
When asked about unrelated topics (CI/CD infra, GPU issues, network config), the model now says so rather than hallucinating an answer.
Instruction Following
Prompts like "write a keyword that opens a browser" now stay focused on the Robot Framework pattern instead of drifting to generic Python.

What's Not Changed Yet

These are areas for the next training iteration: Instruction following is improved (+6pp) but still at 28% — the model sometimes produces correct syntax in the wrong structure. Robot table correctness at 48% leaves room for improvement with more diverse training examples. Both are expected to improve with expanded dataset coverage.
